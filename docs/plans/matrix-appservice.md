# Matrix Application Service for openclaw

## Context

The bot-per-agent plan (see [bot-per-agent.md](./bot-per-agent.md))
gives each openclaw agent its own real Matrix user with its own
device, cross-signing keys, access token, and matrix-bot-sdk
process. That's the right starting shape for ~3-5 agents but
scales poorly: per-agent bootstrap CR registration, per-agent
secrets, per-agent verify dance, per-agent RAM. At 10+ agents
the bookkeeping starts to dominate.

Matrix's spec defines an alternative: an **Application Service
(AS)**. Synapse registers the AS via a YAML config file at
startup; the AS gets:

1. A **user namespace** (regex-matched MXIDs it controls — e.g.
   `@_openclaw_.*:<public_domain>`). No `/login` needed per
   user; the AS asserts identity via its own bearer token
   (`as_token`).
2. A **room namespace** (regex-matched room aliases). Optional.
3. **Push-style event delivery** — Synapse POSTs new events to
   the AS's HTTP endpoint instead of the AS polling `/sync`.

In practice an AS lets one process puppet many ghost users.
That's exactly the "many agents, one process" shape we want.

This plan exists because it's the right *long-term* answer, even
though we shouldn't start here. Capture the design so the future
migration from bot-per-agent → appservice is mechanical.

## Strategy

**Replace the N matrix-bot-sdk processes with one
matrix-appservice process puppeting `@_openclaw_<agent>:<domain>`
ghost users.** All on the same OpenClaw EC2 host as today; just
swap the bot binary.

The single open wound is E2EE. matrix-bot-sdk's appservice
helper does *not* natively support encrypted rooms — ghost users
have no devices, so other clients can't deliver them Megolm
session keys. Three workarounds, listed with their costs:

1. **Run rooms unencrypted.** Defeats the operator's stated
   reason for choosing Matrix as the "safe" interface. Reject.
2. **Use [pantalaimon](https://github.com/matrix-org/pantalaimon)
   as a decrypting proxy.** Pantalaimon sits between the AS and
   Synapse; it has its own client logins (one per ghost user)
   that hold Megolm keys, decrypts incoming events before
   handing them to the AS, encrypts outgoing events on the way
   out. Adds another always-on service to maintain.
3. **Wait for [matrix-rust-sdk's appservice-encryption
   support](https://github.com/matrix-org/matrix-rust-sdk).**
   Synapse 1.119+ has experimental encryption-aware AS support
   gated behind config flags; it isn't a stable surface yet.

For most of 2026, **option 2 (pantalaimon)** is the only working
choice. If matrix-rust-sdk's support matures by the time this
plan is built, switch to option 3.

## Building blocks

### 1. Appservice registration YAML

A file Synapse reads at startup:

```yaml
id: openclaw
url: http://127.0.0.1:9000   # AS HTTP endpoint (same EC2)
as_token: <generated, written to Secrets Manager>
hs_token: <generated, written to Secrets Manager>
sender_localpart: openclaw
namespaces:
  users:
    - exclusive: true
      regex: '@_openclaw_.*:<public_domain>'
  rooms: []
  aliases:
    - exclusive: false
      regex: '#openclaw-.*:<public_domain>'
rate_limited: false
de.sorunome.msc2409.push_ephemeral: false
```

Generated by a Custom Resource at MatrixStack deploy time (mints
the two tokens, writes them to Secrets Manager, uploads the YAML
to EFS where Synapse's `homeserver.yaml` already lives).
Synapse's `homeserver.yaml` references the AS YAML in the
`app_service_config_files:` list.

### 2. Appservice HTTP endpoint

A Node.js (or Python) process listening on `127.0.0.1:9000` and
implementing the AS API:

- `PUT /_matrix/app/v1/transactions/:txnId` — Synapse pushes
  batches of events here. The AS must reply 200; if it doesn't,
  Synapse retries with backoff.
- `GET /_matrix/app/v1/users/:userId` — Synapse asks if a user
  exists in the AS's namespace. The AS responds 200 (create the
  ghost) or 404 (no such user).
- `GET /_matrix/app/v1/rooms/:roomAlias` — same idea for room
  aliases.

matrix-bot-sdk has an `Appservice` class that handles the HTTP
plumbing + event delivery; we'd extend it with our routing
logic (room → agent_id, sender allowlist).

### 3. Pantalaimon shim (only for encrypted rooms)

Pantalaimon runs as a separate process (HTTP server on
`127.0.0.1:8009` or similar):

- Holds N "logins" — one per ghost user. Each login is a real
  `/login` against Synapse using the AS's puppet privilege
  (Synapse's `?user_id=` query param).
- Each login generates a device and cross-signing keys, just
  like a bot user would. Operator runs `bin/matrix-bot-verify`
  once per ghost (same dance as today).
- Pantalaimon transparently decrypts incoming events before
  the AS sees them, and encrypts outgoing events the AS sends.

The AS uses pantalaimon's HTTP API instead of Synapse's
directly. The "appservice → Synapse" path is mediated by
pantalaimon.

### 4. CDK shape (delta from bot-per-agent)

- New `infra/stacks/matrix_appservice_stack.py` (or fold into
  OpenClawStack since they share an EC2 host).
- New `assets/lambdas/matrix_appservice_register/` — the Custom
  Resource that generates `as_token`/`hs_token`, writes the AS
  YAML to EFS, signals Synapse to reload (or restarts the
  Synapse task — easier).
- MatrixStack: add `app_service_config_files` to the init-
  rendered `homeserver.yaml`; persist the AS YAML beside it on
  EFS.
- OpenClaw EC2 user-data: install pantalaimon + the AS binary;
  systemd user units for both.
- Bot code becomes `assets/openclaw_appservice/`, replacing
  `assets/openclaw_bot/`. Rough TS skeleton using
  `matrix-bot-sdk`'s `Appservice` API.

### 5. Migration FROM bot-per-agent

Once the AS is running and verified:

1. Each existing `@openclaw-bot/<agent>` user is **deactivated**
   in Synapse (cleaner than rename — the AS namespace uses
   `@_openclaw_<agent>` to avoid collision).
2. Existing rooms each agent was in: rejoin under the new ghost
   MXID. Old user's `m.room.member` events stay in history but
   the agent isn't in the room anymore under the old MXID.
3. The operator notices their existing DMs go silent under the
   old MXID; new DM invites come from the ghost-namespace MXIDs.
4. Operator deletes the old bot users + cross-signing.json
   files via direct SQL surgery (the same `bin/db-sql` path
   used for ghost-account cleanup in earlier sessions).

This is a one-time disruption per agent — fine for a personal
deployment, painful for a real product.

## Open decisions

- **`@_openclaw_<agent>` vs. cleaner localparts.** Matrix
  convention prefixes AS-puppeted MXIDs with `_<id>_` so they're
  recognizable as ghosts vs. real users. `@_openclaw_wadsworth`
  is unambiguous; `@wadsworth` is prettier. Element shows them
  identically either way; the operator's call.
- **Pantalaimon hosting.** Same EC2 as openclaw daemon? Separate
  Fargate task? Separate EC2? Pantalaimon's resource footprint
  is small (~200 MB RAM), same-host co-location is fine.
- **Whether to fold into existing OpenClawStack** vs. spin off a
  new stack. The AS is conceptually openclaw's matrix-facing
  layer; same lifetime, same EC2; fold in.
- **Wait for matrix-rust-sdk?** matrix-rust-sdk-crypto v1 has
  appservice-encryption hooks; matrix-rust-sdk-appservice
  proper is still in flux. As of writing, pantalaimon is the
  pragmatic answer. Re-evaluate before building.

## Why not start here

- Pantalaimon is a maintenance liability for ~zero gain at 3
  agents. Bot-per-agent at 3 agents is fully functional.
- The migration cost is bounded: one disruption, well-defined,
  scripted.
- matrix-rust-sdk's appservice encryption support might mature
  in the meantime, simplifying option 3 from "experimental
  config flags" to "supported".
- Building the AS now means we'd be reasoning about pantalaimon
  + ghost-user provisioning + encryption proxy + the AS API
  *plus* the agent/room routing logic *plus* openclaw
  integration. That's a lot of new surfaces at once.

The bot-per-agent design pays off the agent-multiplication need
without absorbing the appservice cost. When the per-agent cost
becomes the bottleneck (say, > 10 agents and the per-bot RAM is
biting), we have a clear plan to migrate.

## Verification

- After AS deploy: `curl
  https://matrix.<public_domain>/_synapse/admin/v1/registrations`
  (admin auth) lists `openclaw` in the appservice list.
- `@_openclaw_wadsworth:<public_domain>` (or whatever
  namespace) appears in Synapse's user table without ever
  having gone through `/login`.
- Invite the ghost MXID to a fresh encrypted DM; the AS receives
  the invite event via the `PUT /transactions/:txnId` callback;
  pantalaimon decrypts; AS routes to the wadsworth agent;
  agent responds; pantalaimon re-encrypts; operator sees the
  reply in Element.
- Element shows the ghost user as fully verified (master key
  signed by operator's user_signing — same `bin/matrix-bot-verify`
  flow per ghost).

## Out of scope

- Federation of ghost users to other homeservers. Works
  automatically if Synapse's federation is open, but no
  special infrastructure here.
- Bridging to non-Matrix chat networks (Slack, Discord). The AS
  pattern is also how those bridges are built (e.g.
  matrix-appservice-slack). If we ever want that, the
  infrastructure here is the starting point — but each bridge
  is its own scoped project.
- Multi-homeserver shared agents. One openclaw AS, one Synapse;
  no federation of agent state across homeservers.
